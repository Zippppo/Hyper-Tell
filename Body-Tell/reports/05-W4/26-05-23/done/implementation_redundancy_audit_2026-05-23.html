Body-Tell 实现冗余审查

日期 2026-05-23 · 范围: 静态阅读 Body-Tell/ 与 VoxTell/ 源码, 参考 W&B 训练日志 · 本报告未修改训练实现代码

结论摘要

Body-Tell 的核心模型并没有明显失控。 仅看 Python 源码, Body-Tell/body_tell 是 11 个文件、约 1423 行; VoxTell/voxtell 是 9 个文件、约 1580 行。当前复杂度主要来自训练新数据集所需的 dataset、prompt vocabulary、artifact validation、loss/metric、train loop 和一次性脚本, 而不是模型 forward 本身。

最值得优化的冗余不是大重写, 而是收窄边界。 优先处理 dataset 运行时重复工作、训练脚本职责过宽、配置字段重复映射、未使用的深监督/transformer 分支和 prompt embedding 工具重复。这样可以保持当前训练行为, 同时更接近 VoxTell 的简洁形态。

复杂度来源

范围规模判断
VoxTell/voxtell 9 个 Python 文件, 约 1580 行 主要覆盖模型、推理、text embedding helper。没有训练 loop 和数据集构建链路。
Body-Tell/body_tell 11 个 Python 文件, 约 1423 行 核心包体量与 VoxTell 接近。新增模块多数是迁移到 voxel dataset 后的必要职责。
Body-Tell/train.py 525 行 集中了 CLI、DDP、AMP、W&B、checkpoint、smoke、train/eval loop。这里是最明显的结构性膨胀点。
Body-Tell/scripts 5 个脚本, 约 1121 行 Phase 0 artifact 生成和审计逻辑较多, 但很多是一次性/离线逻辑, 不应污染运行时路径。
Artifact class_presence.json 约 26MB; prompt_embeddings.pt 约 2.7MB; 264 prompt records 预计算 artifact 本身合理, 但 runtime loader 和 repo hygiene 还有精简空间。

优先级位置观察为什么冗余建议
P0 body_tell/data/dataset.py:74-85, body_tell/data/vocabulary.py:248-288 Dataset 初始化先调用 validate_prompt_embedding_cache, 该函数内部 torch.load 一次 cache; 随后 dataset 又 torch.load 同一个 prompt_embeddings.pt。 这是纯运行时重复 I/O 和反序列化, 对行为没有贡献。随着 cache 变大或多 worker/多进程初始化, 成本会放大。 合并为一个 load_prompt_embedding_cache(...), 返回 cache、errors、warnings。严格校验仍保留, 但只加载一次。
P0 body_tell/data/vocabulary.py:291-340, body_tell/data/dataset.py:110-118 sample_prompts_for_case 每次取样都会调用 prompt_records_by_class, 后者每次重新 flatten vocabulary。 训练集 9779 samples, 每个 epoch 都重复构造同一份 prompt 分组。这个逻辑应该是 dataset 初始化期的常量。 在 HyperBodyPromptDataset.__init__ 里预计算 self.prompt_records_by_class; sampler 接收已分组 records 或成为 dataset 私有方法。
P0 body_tell/models/voxtell_body_model.py:83-89, configs/phase1_voxtell_body.yaml:14 配置写 num_maskformer_stages: 5, 但模型用 min(num_maskformer_stages, len(encoder_channels)-1) 静默截断。当前 5 个 encoder stages 实际最多只有 4 个 upsampling/fusion stages。 这是从 VoxTell 迁移时的命名漂移。VoxTell 原始模型有 6 个 decoder configs 和 5 个 decoder stages; Body-Tell 当前 channels 列表只有 5 层。 把字段改成更准确的 num_fusion_stages, 或在请求值超过上限时显式报错/告警。避免配置看起来是 VoxTell 的 5-stage, 实际运行是 4-stage。

P1 models/voxtell_body_model.py:16-30, train.py:110-126, configs/phase1_voxtell_body.yaml:3-16 同一组 model config 字段在 dataclass、YAML 和 build_model 手工映射里重复出现。 添加/重命名字段时需要同步三处, 容易漏掉默认值或新参数。 用 VoxTellBodyConfig(**cfg["model"]) 或 VoxTellBodyConfig.from_dict 统一入口, 保留 unknown-key 校验。
P1 models/voxtell_body_model.py:150-151, losses/prompt_losses.py:70-111, configs/phase1_voxtell_body.yaml:16 当前正式配置 deep_supervision: false, 但模型、loss、测试都维护了一套 deep supervision 分支。 如果 Phase 1 不训练深监督, 这部分是未使用维护面; 如果目标是贴近 VoxTell, 那么它又应该成为正式配置和 loss weights 的一等路径。 做一次明确决策: 要么移除当前未用分支以简化, 要么启用并配置 deep_supervision_weights, 让测试覆盖正式训练路径。
P1 models/transformer.py, VoxTell/voxtell/model/transformer.py Body-Tell 维护了 VoxTell transformer 的本地 fork。它已经简化到 223 行, 但仍保留 post-norm、return_intermediate 等当前配置未使用分支。 本地 fork 容易和 VoxTell 行为漂移。当前正式模型始终 normalize_before=True, 未用分支增加测试和理解成本。 如果继续本地维护, 只保留当前架构使用的 pre-norm decoder。若未来要复用 VoxTell 权重或保持论文实现一致, 则改成共享/拷贝锁定版本并加差异说明。
P1 VoxTell/voxtell/utils/text_embedding.py:3-19, scripts/build_prompt_embedding_cache.py:25-40 last_token_pool 和 wrap_with_instruction 在 VoxTell 与 Body-Tell embedding cache 脚本里重复实现。 这是低风险但明确的重复。instruction 字符串已在 vocabulary.py 中集中, helper 还没有集中。 新增一个很小的 body_tell/text_embedding.py 或 body_tell/data/text_embedding.py, 与 VoxTell 的 19 行 helper 对齐, 脚本只调用它。
P1 body_tell/data/vocabulary.py 同一个模块混合 JSON I/O、hash、vocab validation、embedding cache validation、prompt flattening、sampling、target mask 构造。 运行时 dataset 只需要 prompt records、sampling 和 target mask; build-time 校验和 artifact hash 不应该长期耦合在训练数据读取路径里。 拆成 data/vocabulary.py 只放数据结构和 prompt flatten; data/prompt_sampling.py 放采样; artifacts/validation.py 放离线校验。
P1 scripts/build_label_vocab.py:42-261 70 类 prompt 和 10 个 aggregate 大量写在 Python 源码里。 这让 prompt 内容审查、版本 diff、人工 validation 与代码 review 混在一起。已有 reports/potentail_bug.html 也指出 variants 覆盖和 inside_body_empty 语义风险。 把 prompts/aggregates 移到数据文件, 例如 configs/label_vocab_source.yaml, 脚本只做生成、校验和 hash。这样更接近 VoxTell 的“代码只表达逻辑”。
P2 body_tell/data/dataset.py:97-108 每次 __getitem__ 都从 sensor_pc 重新 voxelize occupancy, 再按 volume size crop/pad。 这是可接受的第一版实现, 但长训练中 CPU 重复工作明显。VoxTell 推理路径通过预处理和 sliding window 把运行时职责分层。 如果 profiling 显示 dataloader 成为瓶颈, 预生成 occupancy 或支持 NPZ 内已有 occupancy 字段。不要在没有瓶颈证据时优先重做。
P2 .gitignore:8-10, tracked artifacts Body-Tell/Dataset/ 和 Body-Tell/artifacts/ 被 .gitignore 忽略, 但已有部分 metadata/artifacts 被 git 跟踪, 例如 class_presence.json。 这会让“哪些是源码, 哪些是可再生产物”不够清楚。26MB 的 class_presence.json 作为源码 review 对象也会制造噪音。 明确策略: 要么保留小型 manifest 并记录生成命令, 要么将大 artifact 移到外部存储。当前不要直接删除, 因为训练依赖它。

建议的目标形态

以 VoxTell 的代码形态为目标时, Body-Tell 可以收敛成四层边界, 而不是把训练、artifact 和模型细节揉在一起。

层级保留职责不应承担的职责
Model VoxTellBodyModel, transformer decoder, mask fusion decoder, dynamic 3D position encoding。 artifact 校验、dataset prompt sampling、训练日志。
Data runtime 读取 NPZ, volume fit, prompt sampling, target mask, collate。 重新扫描全数据集、重建 embedding cache、生成 vocabulary。
Training runtime train_one_epoch, evaluate, checkpoint save/load, metric aggregation。 argparse 细节、W&B 参数解析、artifact 生成。
Offline tools vocab source -> label_vocab, class presence scan, embedding cache build, HTML audit。 被 dataset 每个 sample 间接调用的逻辑。

不建议现在动的部分

  • 保留 sinusoidal_position_encoding_3d 的动态实现。Body-Tell 数据 shape 不是 VoxTell 的固定 DECODER_CONFIGS, 动态 position encoding 是必要适配, 不是冗余。
  • 保留 prompt_valid、negative prompt 和 target_empty 逻辑。它们是 variable prompt count 和 2 positive + 1 negative 训练策略需要的边界。
  • 保留 class_presence.json 作为运行时索引的概念, 但优化加载和归档策略。否则每次训练前扫描全量 voxel labels 会更慢。
  • 保留 现有 tests 作为重构护栏。精简实现前, 这些 shape/loss/dataset/mini-overfit 测试比代码风格更重要。

P0 项架构评审

评审结论 (2026-05-23): 三个 P0 项均值得清理, 但风险等级和实施策略不同。

项目训练安全性收益实施策略
P0-1: 重复加载 prompt_embeddings.pt 零风险 单进程节省一次 2.7MB I/O; 多 worker 场景收益倍增; 代码更清晰 立即可做。新增 load_and_validate_prompt_cache 统一入口, validation 和使用共享同一份数据。
P0-2: 每次采样重新 flatten vocabulary 零风险 消除 9779×30=293,370 次重复计算; 语义更清晰 (常量数据应在初始化时构造) 立即可做。在 HyperBodyPromptDataset.__init__ 预计算 self._prompt_records_by_class, 训练时直接查询。
P0-3: num_maskformer_stages 语义混淆 低风险 (需验证 checkpoint 兼容性) 消除配置与实际行为不一致; 避免未来迁移 VoxTell 权重时的困惑 分两步: (1) 先加 warning 让隐藏截断可见; (2) 确认 checkpoint 加载无影响后, 改配置字段为 num_fusion_stages: 4。

def load_and_validate_prompt_cache(cache_path, vocab):

低风险重构顺序

实施原则: 先做零行为变化的优化, 再碰可能影响模型行为的结构分支。每步都要验证训练正常。

  1. Phase 1: P0-1 和 P0-2 (可合并为一个 PR, 零风险)
    • 统一 embedding cache 加载入口
    • Dataset 初始化时预计算 prompt records by class
    • 验证: 跑 2-3 个 epoch, 确认第一个 epoch 的 loss 值与历史一致
  2. Phase 2: P0-3 第一步 (低风险)
    • 在模型初始化时加 warning, 让 num_maskformer_stages 截断行为可见
    • 验证: checkpoint 保存和恢复正常, warning 符合预期
  3. Phase 3: P0-3 第二步 (需确认 checkpoint 兼容性)
    • 将配置字段改为 num_fusion_stages: 4
    • 如果 checkpoint 保存了 config 并做严格校验, 需要兼容旧字段名或重新训练
  4. Phase 4: P1 项 - 拆训练入口
    • 把 train_one_epoch、evaluate、checkpoint helpers 放进 package
    • 让 train.py 变成薄 CLI
    • build_model 改成 dataclass factory
  5. Phase 5: P1 项 - 处理架构开关
    • 明确 deep_supervision 的正式语义: 要么删未用分支, 要么启用并配置 loss weights
    • 决策 transformer 本地 fork 策略: 只保留 pre-norm 分支, 或改成共享/锁定版本
  6. Phase 6: P1 项 - 离线 artifact 清理
    • 将 prompt source 数据化 (移到 configs/label_vocab_source.yaml)
    • 将 vocabulary validation 从 runtime sampler 拆出
    • 同步解决 potentail_bug.html 中记录的 label 语义和 variants 覆盖问题

验证清单 (每个 Phase 完成后)

  • 训练启动正常, dataset 初始化无报错
  • 第一个 epoch 的 loss 值与历史可比 (Phase 1-3 必须一致)
  • Checkpoint 保存和恢复正常
  • 如果有 validation, val metrics 与历史可比
  • W&B 日志完整, 无异常 warning

最终判断

如果最终目标是 VoxTell 那种简单优雅的代码, Body-Tell 不需要把所有训练适配都压缩到一个模型文件里。更合理的目标是: 模型像 VoxTell 一样直接, dataset 像普通 PyTorch Dataset 一样薄, train.py 像 CLI wrapper 一样薄, 所有一次性 artifact 逻辑留在 scripts 或 reports。

当前最值得先做的优化是 P0 三项: cache 单次加载、prompt 分组预计算、明确 num_maskformer_stages 的实际语义。这三项不需要改变算法, 但会显著减少隐藏冗余和迁移语义漂移。
