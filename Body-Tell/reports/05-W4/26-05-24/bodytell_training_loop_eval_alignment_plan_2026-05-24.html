执行原则

本计划只处理当前代码中已经暴露的闭环契约问题：模型输出、loss、metrics、DDP 聚合和推理构图必须在同一套训练配置下保持一致。

纳入标准：会导致运行时崩溃；会静默错训或错评；会让训练出的 checkpoint 无法被正式推理入口加载。不满足这些条件的事项不进入本轮修复。

当前范围不包含优化器或 scheduler ablation、完整断点恢复、大评估协议扩展、旧训练分支整理。这些事项不影响本计划的正确性目标。

任务清单

编号 任务 影响 实现边界
T1 明确 deep_supervision=true 下的模型输出契约 当前训练/验证 loop 把 model() 返回值同时当成 loss 输入和 metrics 输入。deep_supervision=true 时，模型返回的是多尺度 supervision bundle；loss 需要完整 bundle，metrics 只能评估主尺度 prediction tensor。现有代码直接对 bundle 调用 logits.float()，暴露出的是运行时崩溃，根因是输出契约没有在 train/evaluate 中被明确拆分。 将变量语义拆开：model_outputs 传给 criterion，保留完整 deep supervision loss；primary_logits(model_outputs) 传给 metrics。primary_logits() 的契约是：Tensor 原样返回；list/tuple 返回第一个主尺度输出，并要求它与 target_masks 同形状，否则 fail-fast。禁止在 loss 前截断为单尺度，也不把 deep supervision bundle 的处理逻辑塞进 metrics 函数。
T2 修正 prompt metrics 的 foreground Dice 聚合 Foreground Dice 只对 positive prompt 有定义。当前 patch-level 采样可能产生没有 positive prompt 的 batch；此时 compute_prompt_metrics() 返回 nan，而训练/验证 loop 会按普通数字累加，进而污染 epoch 指标和 best checkpoint 判断。这不是外部误用防护，而是当前采样机制下需要处理的指标聚合边界。 Foreground Dice 按有效 positive prompt 的 sum/count 聚合；无 positive 的 batch 不贡献 foreground Dice，但仍统计 negative FP。可顺手保留 target_empty 与 prompt_valid shape 校验作为 fail-fast 检查，但它不是本任务的核心目标。
T3 统一训练与推理的模型构建契约 训练入口会把 backbone、n_blocks_per_stage、encoder_conv_bias、encoder_norm、encoder_activation 传给模型；推理入口漏传这些字段。当前 aligned config 使用 residual_encoder，训练出的 checkpoint 进入推理时会用不同结构构图，再被 strict=True 加载，存在正式推理失败风险。 让 Body-Tell/inference.py::build_model() 与 Body-Tell/train.py::build_model() 字段保持一致，或抽出共享构建 helper。验收以同一 config 构建出的关键结构字段一致为准。
T4 DDP 指标聚合后再日志与保存 best 当前正式脚本使用 torchrun 多 GPU，train/val 指标却是 rank-local 均值；rank0 保存 best checkpoint 也只看 rank0 的验证 shard。这会让长训曲线和 best checkpoint 选择不代表全体验证集，是静默实验错误。 对 loss、BCE、Dice、positive Dice 分子/计数、negative FP 分子/计数做 all-reduce 后再产出日志、W&B 和 best 判断。不引入 fixed-update 或 scheduler。

实施验证准则

本计划按 TDD 执行：每个任务先补能够暴露当前问题的针对性验证，再实现修复。验证名称和脚本形式不强制统一，但必须能证明该任务的根因被修复，而不是只证明某个表面报错被绕过。

硬性训练 smoke：每一次代码改动完成后，都必须运行 h20b-sh-scripts/test-train.sh，使用专门的 Body-Tell/configs/test-model-config.yaml 检查训练是否能正常启动和进入训练流程。不要求等待完整训练结束，但必须确认不是在 config 加载、dataset 构建、模型构建、loss/metrics 或第一轮训练前向/反向处立即失败。

不要把 test-train.sh 生搬硬套成不可变脚本。它的职责是验证真实训练入口在本轮改动后仍然跑得通；如果代码改动改变了 config 契约、模型构建字段、loss/metrics 参数或训练入口参数，必须同步维护 Body-Tell/configs/test-model-config.yaml，让 smoke 覆盖新的真实契约，而不是让测试配置停留在旧接口上。

因此，任务级 pytest 只负责精确锁定 bug；test-train.sh 负责确认真实训练入口仍然可运行。两者不能互相替代。
