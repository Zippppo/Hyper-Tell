动态 HWD 位置编码与 Transformer Decoder 权重复用说明

生成日期：2026-05-28。范围：解释 Body-Tell 中 runtime HWD position encoding 的设计目的、张量契约、与 VoxTell transformer_decoder.* 权重复用的关系。

一句话结论权重复用的输入契约对齐层
不是单独的训练增强技巧
核心张量顺序B,C,D,H,W -> H*W*D,B,C
中间按 B,H,W,D,C 排列
复用边界加载 decoder, 跳过 pos_embed
P6-lite 只复用 transformer_decoder.*

结论

这个点可以理解为：为了让之后的 VoxTell transformer_decoder.* 权重复用不只是“key 和 shape 对得上”，而是让 decoder 看到的 image memory 在 token 顺序、空间位置、dtype 和 device 上都满足同一套输入契约。

它更准确的定位是 input-contract alignment。如果没有这层对齐，即使 strict load 成功，VoxTell 的 cross-attention 权重也可能在 Body-Tell 里处理错位的 3D token 序列。

因此，它主要提升的是结构兼容性、运行稳定性和权重复用可信度。它不是像 foreground oversampling 或 prompt 变体那样直接改变训练分布的正则化技巧。

为什么要做这个改动

问题 如果不处理会怎样 当前做法
VoxTell checkpoint 中的 learned pos_embed 隐含固定 token 网格。 Body-Tell 当前 whole case 是 128x128x256，训练 patch 是 128x128x128，经过 encoder 后的 bottleneck token 网格不等同 VoxTell 固定网格。强行加载可能 shape 不匹配，或需要危险插值。 P6-lite loader 明确把 pos_embed 放进 excluded_prefixes，不复用固定位置编码。
Transformer decoder 的 cross-attention 需要 image memory 序列和位置编码。 如果 image token 的 flatten 顺序不同，decoder 权重即使加载成功，也会把某个空间点误解成另一个空间点。 Body-Tell 在 forward 中按 B,C,D,H,W -> B,H,W,D,C -> H*W*D,B,C 生成 memory，并按同样 HWD 顺序生成 runtime 位置编码。
AMP 或 CUDA 下 dtype/device 必须一致。 位置编码和 memory 的 dtype/device 不一致，会导致运行时错误，或触发隐式 cast，增加不稳定性。 runtime position encoding 使用 memory.device 和 memory.dtype，随后强校验 shape、dtype、device。

具体张量流程

1. Encoder 输出selected_feature
B,C,D,H,W
2. 通道后置permute(0,3,4,2,1)
B,H,W,D,C
3. 投影到 query dimproject_bottleneck_embed
C -> 2048
4. 展平为 memoryH*W*D,B,2048
供 cross-attention 使用
5. 生成 possinusoidal_position_encoding_3d(H,W,D)
shape 与 memory 完全一致
batch_size, channels, depth, height, width = selected_feature.shape

bottleneck_embed = selected_feature.permute(0, 3, 4, 2, 1)
bottleneck_embed = self.project_bottleneck_embed(bottleneck_embed)
memory = bottleneck_embed.permute(1, 2, 3, 0, 4).reshape(
    -1,
    batch_size,
    self.config.query_dim,
)

pos = sinusoidal_position_encoding_3d(
    (height, width, depth),
    self.config.query_dim,
    device=memory.device,
    dtype=memory.dtype,
    batch_size=batch_size,
)

关键点是：虽然 PyTorch 3D feature 原始顺序是 D,H,W，这里进入 transformer 的序列顺序使用 H,W,D。这样做是为了和 VoxTell transformer memory 的排列语义对齐。

它和权重复用的关系

VoxTell transformer_decoder.* 权重中，cross-attention 的核心计算大致是：

query = text prompt embedding
key   = image memory + position encoding
value = image memory

所以权重复用需要满足下面这几层契约：

契约层 当前是否处理 对应实现 意义
参数 key 与 shape 已处理 load_voxtell_transformer_decoder_prefix() 严格检查 missing、unexpected、shape mismatch。 保证 checkpoint tensor 能合法装入 Body-Tell transformer decoder。
image memory embedding dim 已处理 project_bottleneck_embed 把 selected feature 投影到 query_dim=2048。 保证 image key/value 维度和 decoder 权重期望一致。
text query embedding dim 已处理 project_text_embed 把 Qwen 2560 维文本投影到 2048。 保证 prompt query 维度和 decoder 权重期望一致。
image token flatten 顺序 已处理 B,C,D,H,W -> B,H,W,D,C -> H*W*D,B,C。 保证 decoder 权重看到的序列顺序和 VoxTell 语义一致。
position encoding shape 与语义 已处理 按真实 (height,width,depth) runtime 生成 sinusoidal 3D position。 避免固定 pos_embed 对 Body-Tell 非立方输入造成错配。
device 与 dtype 已处理 使用 memory.device、memory.dtype，并 fail-fast 校验。 支持 CUDA/AMP 下稳定运行。

所以它确实是为 transformer decoder 权重复用做准备，但准备的不是“加载代码”本身，而是 加载之后 forward 输入语义仍然成立。

为什么跳过 fixed pos_embed 更合理

方案优点风险当前选择
直接加载 VoxTell fixed pos_embed 保留 VoxTell learned spatial prior。 依赖固定 token 网格。Body-Tell 的 128x128x256 和 patch 输入可能无法对齐，shape 或空间语义都可能错。 不采用
对 fixed pos_embed 插值 可以勉强适配不同 shape。 非立方输入和 HWD/DHW 顺序容易引入隐性空间错位；一旦 silent interpolation 错了，debug 成本高。 不采用
runtime sinusoidal 3D position 任意当前 feature shape 都能生成匹配位置编码；无需加载固定网格；shape/dtype/device 可强校验。 不复用 VoxTell learned pos_embed 本身的空间先验。 当前采用

如果没有这个对齐层

  1. 最直接的失败：pos_embed shape 对不上，加载或 forward 直接报错。
  2. 更危险的失败：shape 通过插值或 reshape 勉强对上，但 token 顺序和位置语义错了，模型能跑但 cross-attention 学到的是错位空间关系。
  3. 迁移收益被削弱：VoxTell transformer decoder 的权重原本学到的是 text query 如何 attend 到 image memory。如果 memory 的空间排序不一致，权重名义上复用了，语义上并没有真正复用。
  4. 混精和多设备更脆弱：位置编码如果没有跟随 memory 的 dtype/device，AMP 或 CUDA 下容易出现运行错误。

最终表述建议

在总报告或论文式总结里，建议这样表述：

Dynamic HWD position encoding is introduced as an input-contract
alignment layer for VoxTell transformer-decoder prefix reuse.
It does not reuse VoxTell's fixed learned pos_embed. Instead, Body-Tell
flattens the runtime 3D feature map in the VoxTell-compatible H,W,D order
and generates a shape-, dtype-, and device-matched 3D sinusoidal position
encoding. This makes strict transformer_decoder.* weight reuse semantically
safer for Body-Tell's non-cubic 128x128x256 volumes and 128x128x128 patches.

中文短句可以写成：动态 HWD position encoding 是为了让 VoxTell transformer decoder 的权重复用具备正确的 memory/position 输入语义，而不是为了复用 fixed pos_embed 本身。
