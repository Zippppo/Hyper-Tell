Body-Tell Encoder Backbone 对比报告

生成日期：2026-05-23；范围：Body-Tell 当前实现 vs VoxTell v1.1 plans / model code。

核心结论

Body-Tell 总参数328.98M
VoxTell 总参数440.03M
Body-Tell encoder8.49M
VoxTell encoder90.30M
encoder 参数倍率10.63x
全模型参数倍率1.34x

Body-Tell 的 encoder 确实是轻量版：它不是 nnU-Net 风格的 ResidualEncoder，而是 5 个普通 3D 卷积块组成的 _BodyEncoder。与 VoxTell 的 6-stage residual backbone 相比，encoder 参数少约 81.80M，单看 encoder 是非常明显的差别。

但从全模型看，差距没有 10 倍那么夸张，因为两边都保留了同量级的文本投影和 6 层 transformer decoder。Body-Tell 当前最大的参数来源仍然是 transformer_decoder，约 251.86M。

如果当前训练表现为模型难以学到稳定的空间/器官结构，encoder 成为瓶颈是一个高优先级假设。Body-Tell 的 decoder 和 prompt 模块很重，但 visual backbone 只有浅层非残差 CNN；这会让文本侧容量远大于视觉侧容量，模型可能缺少足够强的 occupancy-to-anatomy 表征。

代码中没有找到直接解释“为什么选轻量 encoder”的注释或设计文档。因此本报告把“为什么”分为两层：一是代码事实；二是基于输入模态、显存/计算和迁移阶段的工程推断。

ResidualEncoder 逐源码核实

本次已经从三层证据交叉确定 VoxTell 的 nnU-Net-style encoder 结构：VoxTell v1.1 plans.json、VoxTell model 构造代码、以及当前环境安装的 dynamic_network_architectures 源码。结论是：VoxTell 使用的不是一个普通 CNN encoder，而是 ResidualEncoder 包装的 6-stage residual feature pyramid。

组件源码行为VoxTell v1.1 中的实际结果
ResidualEncoder 如果 disable_default_stem=False，先创建一个 StackedConvBlocks(1) stem；随后循环 n_stages 次，每个 stage 创建一个 StackedResidualBlocks；forward 中依次执行 stem 和 stages，并把每个 stage 输出放入 ret，return_skips=True 时返回全部 skip。 VoxTell 构造时传入 return_skips=True、disable_default_stem=False，因此 decoder 和 prompt fusion 使用的是 6 个多尺度 skip feature。
默认 stem StackedConvBlocks(1, Conv3d, input_channels, stem_channels, kernel_size, stride=1)；stem 不做下采样。 实际实例化为 Conv3d(1,32,k=3,s=1,p=1) + InstanceNorm3d + LeakyReLU，参数 960。
StackedResidualBlocks 第一个 residual block 使用 stage 的 initial_stride；同一 stage 内后续 block 都是 stride=1。n_blocks_per_stage 决定每个 stage 的深度。 VoxTell 的 block 数是 [1,3,4,6,6,6]，总计 26 个 BasicBlockD。
BasicBlockD 主路径 conv1 = ConvDropoutNormReLU(... stride=stage_stride ...)；conv2 = ConvDropoutNormReLU(... stride=1, nonlin=None ...)；之后 out += residual，再执行 nonlin2。 在 VoxTell plans 下，每个主路径卷积是 Conv3d + InstanceNorm3d，第一层带 LeakyReLU，第二层不带激活，残差相加后再 LeakyReLU。
BasicBlockD skip 路径 如果 stride 不为 1，先用 AvgPool3d(stride)；如果输入/输出通道不同，再接 1x1 Conv + Norm；如果 stride 和通道都不变，则 skip 是 identity。 stage 1-4 的首个 block 是 AvgPool3d + 1x1 projection；stage 5 的首个 block 因为 320->320 通道不变，所以 skip 只有 AvgPool3d。
ConvDropoutNormReLU 卷积使用 same padding 逻辑：padding=(kernel_size-1)//2；随后按配置接 dropout、norm、nonlin。 VoxTell plans 设置 Conv3d、InstanceNorm3d(affine=True)、LeakyReLU(inplace=True)、conv_bias=True、无 dropout。

结构差异

维度Body-Tell 当前实现VoxTell / nnU-Net-style 实现影响
输入模态 binary occupancy grid，来自表面点云体素化。 CT/MRI/PET 等 dense medical image patch，经过 z-score normalization。 occupancy 没有 HU/intensity texture、跨模态强度差异和图像噪声模式；图像 backbone 需要更强的纹理和尺度建模能力。
stage 数 5 stages：[32,64,128,256,320]。 6 stages：[32,64,128,256,320,320]。 VoxTell 多一个低分辨率 stage，感受野更深，参数和 activation 也更高。
每 stage 深度 每 stage 一个 _ConvBlock，每个 block 两个 3D conv。 stage block 数 [1,3,4,6,6,6]，总计 26 个 residual blocks，另有 stem。 VoxTell 的高层 stage 特别深，主要参数差距来自 stage 3-5。
残差连接 无 residual skip。 BasicBlockD residual path；stride 或通道变化时用 avgpool + 1x1 projection skip。 残差结构让更深的 3D encoder 更容易训练；Body-Tell 的浅层普通 CNN 更简单但表达能力较弱。
归一化/激活 GroupNorm + GELU，卷积 bias=False。 InstanceNorm3d(affine=True) + LeakyReLU，conv_bias=True。 VoxTell 更接近 nnU-Net 医学图像分割默认风格；Body-Tell 更像轻量自定义 3D CNN。
decoder 契约 decoder 只接收 encoder_channels，与 nnU-Net encoder 解耦。 decoder 直接依赖 encoder 的 output_channels、strides、kernel_sizes。 Body-Tell 更容易脱离 VoxTell/nnU-Net 依赖运行，但也失去 plans 驱动的一致性。
mask projection stage 配置请求 5，但代码用 min(num_maskformer_stages, len(encoder_channels)-1)，实际只有 4 个 projection。 5 个 projection，对应 VoxTell 的前 5 个 decoder configs。 Body-Tell 除 encoder 变浅外，多尺度 mask fusion 也少一级。

为什么 Body-Tell 可能被写成轻量 encoder

  1. 输入信号更简单。Body-Tell 的输入是 binary occupancy，表示表面点是否落入体素；VoxTell 的输入是 CT/MRI/PET dense intensity volume。医学图像 encoder 要处理强度、纹理、噪声、器官边界和跨模态差异，因此 nnU-Net-style residual encoder 更合理。occupancy grid 没有这些强度统计，第一阶段用轻量几何 CNN 是可以理解的。
  2. 3D 体素计算成本非常敏感。Body-Tell 配置的 volume size 是 [144,129,256]。如果直接搬 6-stage residual encoder，不只是多 81.8M encoder 参数，还会显著增加 3D convolution activation、显存和训练时间。参数量只是下限，实际 compute/FLOPs 差距会更大。
  3. 当前模型的瓶颈不只在 encoder。Body-Tell 即使用轻量 encoder，总参数仍有 328.98M，其中 transformer decoder 独占 251.86M。继续加重 encoder 会让全模型接近或超过 VoxTell 的 440M，同时增加过拟合风险。
  4. 迁移实现更少依赖。Body-Tell 当前没有直接依赖 nnU-Net plans 构建 encoder/decoder，而是手写 _BodyEncoder 和 _VoxTellBodyDecoder。这降低了迁移复杂度，也让模型能围绕 occupancy grid 快速跑通训练。

这些是基于代码和任务差异的工程推断，不等同于作者显式设计说明。仓库内没有找到“轻量 encoder”设计 rationale 的文字记录。

这是否一定合理

轻量 encoder 是合理的 baseline，但不一定是最终最优。当前 Body-Tell 的 encoder 只有 8.49M 参数，占全模型约 2.58%；如果验证 Dice 长期不动，问题可能来自 prompt/label 分布、数据划分、occupancy 表达的信息缺失，也可能来自 visual encoder 表达能力不足。

保留轻量 encoder 的理由

  • 更低显存和计算开销，适合快速迭代。
  • occupancy grid 没有医学图像强度纹理，未必需要完整 nnU-Net encoder。
  • 可以减少在当前数据规模上的过拟合压力。
  • 实现简单，便于定位 prompt-conditioned decoder 和 loss 的问题。

潜在风险

  • 5-stage 浅 encoder 可能不足以建模长距离人体结构关系。
  • 没有 residual block，深层表达和梯度路径弱于 VoxTell。
  • 实际 fused mask stage 只有 4 个，少于 VoxTell 的 5 个多尺度融合点。
  • 如果 occupancy 很稀疏，浅层 CNN 可能更难把表面点转成稳定体部语义。

建议

  1. 第一优先级：做 5-stage residual-lite encoder。保留 Body-Tell 当前通道 [32,64,128,256,320]、stride 规则和 decoder 接口，只把每个 _ConvBlock 替换成 residual stage。建议先试 n_blocks=[1,1,1,1,1] 或 [1,2,2,2,2]，这样最小化改动，同时直接验证“没有 residual/encoder 太浅”是否是瓶颈。
  2. 第二优先级：引入 plans-like ResidualEncoder，但仍保持 5 stages。可以使用 dynamic_network_architectures 的 ResidualEncoder 或本地复刻 BasicBlockD 思路；输入仍是 occupancy，norm/activation 可先保持 Body-Tell 的 GroupNorm + GELU，避免一次改变过多变量。
  3. 第三优先级：测试 VoxTell 6-stage 形态。增加第 6 个 320 stage 会改变 selected feature 分辨率、decoder stage 数和 mask projection/fusion 数，应该作为单独实验，不应和 residual 化同时混在一个 ablation 里。
  4. 不要把完整 VoxTell encoder 直接当作默认替换。仅 encoder 就会从 8.49M 增到 90.30M；全模型预计增加至少 81.80M 参数，且 activation/FLOPs 增幅更敏感。直接替换会让训练变慢，也更难判断收益来自残差、深度、第 6 stage 还是 norm/activation。
  5. 修正或确认 mask stage 语义。当前配置写 num_maskformer_stages=5，但实际 fused_stage_count=4。如果 encoder 改成 6-stage，这个问题会直接影响多尺度融合是否对齐 VoxTell。
  6. 如果目标是复用 VoxTell 权重，encoder 仍不是低成本入口。Body-Tell 输入 occupancy 与 VoxTell 图像输入语义不同，且 stage 数不同。相比 encoder，text projection、transformer decoder、mask projection 的结构兼容性更值得优先检查。
