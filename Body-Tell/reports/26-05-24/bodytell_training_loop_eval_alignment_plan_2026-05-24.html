本文件的角色

SP-C 只负责训练控制面与评估协议，不负责模型结构迁移，也不负责数据采样器和 vocabulary 扩充。结构问题继续由 SP-A 负责，数据构造问题继续由 SP-B 负责。

SP-C 的任务是把父报告中已经确认的训练侧差异变成可落地步骤：固定 update-count 训练节奏、SGD + polynomial decay、5-scale deep supervision、可精确恢复的 checkpoint、prompt-aware 评估协议，以及能暴露这些行为的测试矩阵。

核心澄清：Task 0 和 Task 1 不是直接提升 Dice 的训练技巧，而是训练控制面改造：先把预算、step 语义、日志、scheduler 横轴和 resume 语义固定下来，后续 SGD / poly / deep supervision / prompt 评估才有可比较的实验口径。性能收益只能通过后续 ablation 和回归闸门验证，不能从“更像论文”自动推出。

本计划不把当前默认训练全部强制切到 paper-mechanics profile。先新增显式 profile 和代码路径，保留当前 AdamW/短训配置作为回退实验；只有在 smoke、quick val 和 resume 验收通过后，才应把长训入口切到该 profile。

实施任务

Task 0. 新增 paper-mechanics profile 骨架（非论文复刻）

  1. 新增 Body-Tell/configs/phase1_voxtell_paper_aligned.yaml。文件名可沿用 paper_aligned，但配置字段和启动日志必须显式声明这是 paper_mechanics_aligned / non_replica profile，避免误读为论文 500k step 复刻。
  2. 集中写入训练机制：iterations_per_epoch: 250（固定每 epoch 的 optimizer update 数）、optimizer: sgd、lr: 1e-4、lr_schedule: poly、deep_supervision: true、deep_supervision_weights: [1, 0.5, 0.25, 0.125, 0.0625]。这些字段定义的是可比较实验口径，不代表单独启用 Task 0 就会改善性能。
  3. 训练预算字段 2026-05-24 锁定，不写论文 2000 / 500000。profile 必须直接写入下列具体数字：
    • batch_size: 2（per GPU）
    • num_gpus: 2
    • epochs: 75
    • iterations_per_epoch: 250（机制对齐保留）
    • 派生：global_batch: 4、total_updates: 18750、paper_step_ratio: 0.0375
    必须以独立字段 compute_budget 块写入这些值（不要散落在 dataloader / optimizer 子段），便于日志和 W&B 一次性打印。不允许把 epochs 或 total_updates 隐式从其他字段倒推——值就是 75 和 18,750，写死即可；实现侧仍必须 assert epochs * iterations_per_epoch == total_updates，防止配置漂移。详见父文档 总控计划 中 2026-05-24 决议。
  4. 模型字段应引用 SP-A aligned 架构：architecture: voxtell_v1_1、num_maskformer_stages: 5。SP-A P0-P6-lite 已完成；该 profile 可以直接引用 aligned 架构，但默认训练入口仍应等 smoke / resume 验收后再切换。
  5. 训练启动日志和 W&B config 中打印 paper_mechanics profile、compute_budget 全字段（含 total_updates=18750 与 paper_step_ratio=0.0375）、optimizer/scheduler、deep supervision 权重。日志一行 banner 显式提示："训练预算 18,750 step（bz=2×2, ep=75），约论文 500k 的 3.75%，非论文复刻"。

验收：读取配置时能区分当前 baseline profile 与 paper-mechanics profile；compute_budget 字段值与父文档锁定一致（epochs=75、total_updates=18750），且一致性 assert 通过；未启用 SP-A aligned 架构时，长训入口不能静默落回旧轻量模型。Task 0 只验收配置与日志控制面，不以 Dice 变化作为通过条件。

Task 1. 固定 update-count 训练 loop（global_step 是训练进度）

  1. 新增 iterations_per_epoch 支持；当该字段存在时，一个 epoch 固定执行指定数量的 optimizer updates，不再等价于完整遍历 DataLoader。训练长度由 global_step / total_updates 定义，epoch 只作为 sampler seed、日志、评估周期和 checkpoint 周期单位。
  2. 实现可复用的无限 dataloader iterator；迭代耗尽后重建 iterator，但 epoch、iteration_in_epoch 和 global_step 语义保持连续。
  3. 日志同时记录 epoch、iteration_in_epoch、global_step、total_updates、samples_seen。
  4. 每个 epoch 调用 DistributedSampler.set_epoch(epoch)；如果 dataset 提供 set_epoch(epoch)，同步调用它，保证 SP-B 动态 prompt RNG 能随 epoch 更新。
  5. （2026-05-24 修订）smoke 模式语义修正：当前 train.py:365 是 epochs = 200 if smoke else 50（反直觉，smoke 反而跑更多）。重写为 epochs = 2 if smoke else profile.epochs，并允许 --iterations-per-epoch CLI 覆盖；smoke 走同一 fixed-update 代码路径。

验收：2-epoch smoke 的 optimizer step 数严格等于 epochs * iterations_per_epoch；日志中的 global_step 单调递增且无跳号；smoke 默认 epoch 数 ≤ 2（修复反直觉默认）。Task 1 只保证训练长度和采样时钟可控，不直接承诺性能提升。

Task 2. 配置驱动 optimizer profile

  1. 把 optimizer 构造从硬编码 torch.optim.AdamW(... weight_decay=1e-5) 改为配置驱动 factory。
  2. paper-mechanics profile 使用 torch.optim.SGD，初始 LR 为 1e-4。SGD 的 momentum、weight_decay 必须来自配置并写入日志；如果无论文/官方依据，不应把 AdamW 的 weight decay 直接迁移过来。
  3. baseline profile 保留 AdamW，避免无法回退或无法做 ablation。
  4. checkpoint 中写入 optimizer 名称、param group 数、各 group 的初始 LR 与当前 LR。

验收：同一训练脚本能通过配置切换 AdamW 与 SGD；paper-mechanics profile 启动日志中明确显示 optimizer=sgd、lr=1e-4。短预算下 SGD 不预设优于 AdamW，必须保留同预算 ablation。

Task 3. Polynomial LR schedule

  1. 新增 poly scheduler，按 optimizer update 数计算进度：progress = global_step / total_updates。
  2. LR 在每次 optimizer step 后更新，而不是每个 epoch 后更新。
  3. poly_power 进入配置并写入日志；若采用 nnU-Net 惯例值，应在配置中显式声明，避免隐藏默认。
  4. scheduler state 必须能从 checkpoint 恢复；恢复后同一 global_step 的 LR 应与不中断训练完全一致。

验收：scheduler 数值单测覆盖 step 0、半程、末程和 resume 后一步；LR 不应在 epoch 边界出现额外跳变。18,750 step 短预算下，poly decay 是否优于 constant LR 仍属于实验问题。

Task 4. Checkpoint 与精确恢复

（2026-05-24 修订）当前现状交底，避免误读：
  • 当前 best.pt 只写 {epoch, model_state_dict, optimizer_state_dict, metrics, config}；最终 last.pt 也只覆盖训练结束状态。两者都不包含 scheduler state（因为根本没有 scheduler）、不包含 AMP scaler state（虽然 train.py:25, 438 已使用 GradScaler，但 scaler.state_dict() 未持久化）、不包含 global_step / iteration_in_epoch / architecture tag。
  • 当前没有任何周期性、resume-safe 的 latest.pt。训练中断后不能从任意中断点续训；从 best.pt 或最终 last.pt 重新开始都会导致 LR、epoch counter、scaler 和 sampler 时钟错位。
  1. checkpoint 写入 model_state_dict、optimizer_state_dict、scheduler_state_dict、epoch、iteration_in_epoch、global_step、total_updates、scaler.state_dict()（新增，对应已存在但未持久化的 GradScaler）、配置 hash、architecture 字段。
  2. 每 N 次 update（默认 N = iterations_per_epoch）写入 latest.pt；best.pt 保留现有"Dice 提升时写入"语义。两者共存，不互相覆盖。
  3. 恢复入口必须在 dataloader / sampler / dataset epoch state / scheduler state / scaler state 全部恢复后继续训练；如果验收要声称 interrupted 与 uninterrupted 在 batch 序列上严格等价，还必须持久化 Python / NumPy / Torch / CUDA RNG state，否则只能称为 step/LR/scaler 连续恢复。
  4. 恢复日志打印 checkpoint 中的 global_step、下一个 epoch / iteration_in_epoch、当前 LR、配置 hash 是否匹配。

验收：构造 interrupted run 与 uninterrupted run，在同一 global_step 比较 LR、scheduler state、scaler scale 值，必须完全一致；若保存 RNG state，则同一 step 的下一批次也应一致。latest.pt 能在任意中断点恢复并继续单调 global_step，且不覆盖 best.pt / last.pt 的现有语义。

Task 5. 启用 5-scale deep supervision loss + BCE/Dice 公式 audit（含 #12）

  1. SP-A P3 前置已满足：模型必须返回 5 个尺度输出，且最高分辨率输出仍与 target mask 的轴语义一致。
  2. paper-mechanics profile 设置 deep_supervision: true 与权重 [1, 1/2, 1/4, 1/8, 1/16]。
  3. loss 层保存原始权重，并在日志中同时报告归一化后的有效权重；避免“论文权重”和“实现归一化”混淆。
  4. 每个 scale 的 target 使用统一 resize 策略；occupancy grid 小目标在低分辨率 target 中可能被 nearest resize 消掉，日志至少记录总 loss、BCE、Dice、per-scale loss，以及每个 scale 的 positive/empty target 计数。
  5. 补充反传测试：5 个尺度输出都参与 loss，且每个可训练输出分支都有梯度。
  6. （2026-05-24 新增，覆盖 #12，域差异豁免）BCE+Dice 公式 audit（不强制对齐）：对照 nnU-Net DC_and_BCE_loss（或 VoxTell 引用的等价实现）的关键字段——BCE 与 Dice 权重比、smooth 常数、batch_dice、do_bg、ignore_label 处理、reduce 策略——在 PromptSegmentationLoss 内逐项 audit 并列出偏差表。对齐不是强制目标：nnU-Net 默认值是为医学影像（强度分布、噪声、稀疏前景）调过的；occupancy grid 任务下，Body-Tell 当前公式可能已经更合适。处理策略：(a) audit 后偏差小且对齐成本低 → 对齐；(b) 偏差大或对齐后 smoke 指标下降 → 保留 Body-Tell 现公式并把偏差表写入 audit 记录；(c) 不能仅因"和论文不一样"就改公式。

验收：1-batch backward 通过；deep supervision 打开时返回 5-scale loss，关闭时仍能跑 baseline；paper-mechanics 长训仍必须等待 SP-C Task 4、已验收的 SP-B patch/concept sampler 口径和端到端 smoke，而不是仅凭 SP-A 完成即标记通过；BCE+Dice 公式 audit 报告（即使结论是"保留现状"）写入 SP-C 子目录，所有偏差项均有显式记录与处理判断（对齐 / 保留 / 待定）。

Task 6. 评估协议扩展（含 baseline 回归闸门）

  1. 保留现有 S2I val foreground mean Dice 与 negative FP，作为本地域内基线。
  2. 新增 prompt variants 评估：同一 class 用 canonical 与 variant prompt 分组统计 Dice。
  3. 新增 negative empty-mask 评估：对 patch/case absent prompt 统计 FP voxel、FP rate、empty-mask success rate。
  4. 新增 held-out class/concept 分组评估；如果当前 split 没有正式 held-out concept，先用配置化列表声明，不在报告中伪造论文级 zero-shot 结论。
  5. 临床 report-derived prompts 与 ReXGroundingCT fine-tuning 只登记为论文协议缺口；在 Body-Tell 缺少对应数据前，不作为本轮强制验收。
  6. （2026-05-24 新增）Baseline 回归闸门：aligned random / aligned full 任一变体在 2 epoch smoke + quick val 后，val_foreground_mean_dice 必须 ≥ 当前轻量 baseline 的 70%（约 0.3257 × 0.7 = 0.228），否则触发回退并复盘 SP-A / SP-B 已落地改动。当前 --smoke 语义会跳过 val，因此该闸门必须显式启用 quick val 或使用单独 smoke-eval 命令；该阈值仅用于"明显回归"识别，不是性能目标。

验收：评估报告能按 known concepts、prompt variants、negative prompts、held-out classes/concepts 分组输出 Dice/FP，并记录使用的 prompt records / vocab hash；baseline 回归闸门必须在 paper-mechanics 长训启动前完成评估。

落地顺序

  1. 先做 Task 0，让 paper-mechanics profile 有显式配置入口和非复刻预算声明，但不改变默认训练。
  2. 随后做 Task 1、Task 2、Task 3，形成 fixed-update + SGD + poly 的最小训练闭环。
  3. 再做 Task 4，确保中断恢复不会破坏 poly schedule。
  4. SP-A P3 与 SP-B Step 1/2/4 已完成；Task 5 等 SP-C Task 4 后执行。Task 6 可基于 patch-level sampler 与 aggregate metadata 验证 prompt/negative 口径，但最终 vocabulary/cache 一致性需等 SP-B Step 6 重建 cache、再由 SP-B Step 5 rerun Slurm gate 通过。
  5. Task 7 贯穿每个阶段：每完成一项机制，就补对应测试，避免把长训当成唯一验收。

SP-C 的关键路径是 Task 0 → Task 1/2/3 → Task 4 → Task 5。Task 6 可先基于已验收的 SP-B patch-level 与 aggregate concept sampler 做 prompt/negative 评估框架，但最终指标必须等 SP-B Step 6 cache rebuild 完成，并由 SP-B Step 5 rerun phase1 Slurm gate 通过后再定稿。

最终判定

SP-C 已接管父报告中所有训练控制面与评估协议缺口。SP-A 已提供 aligned model 与 P6-lite 权重加载入口，SP-B Step 1/2/4 已提供动态采样 RNG、patch-level 2+1 prompt 口径与 aggregate concept sampler；SP-B Step 5 当前 gate_failed，SP-B Step 6 已按 dependency exception 准备先重建 stale embedding cache。SP-C 现在要补齐 optimizer/scheduler、deep supervision、checkpoint resume 与评估协议，形成可验证的 paper-mechanics 训练闭环。
