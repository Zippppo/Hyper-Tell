# Body-Tell Agent Workflow

This directory turns the existing four HTML plans into an executable workflow for long-running Codex work.

## Files

- `lead.html`: human-readable lead plan and final progress log. Only the manager may edit it.
- `workflow_manifest.yaml`: machine-readable task graph, owners, dependencies, status, and acceptance criteria.
- `LEADER_PROMPT.md`: prompt for running one interactive Codex CLI session as the leader.
- `workflow_status.json`: compact leader-safe machine status generated by `run_manager.py status` or `monitor`.
- `workflow_status.html`: compact leader-safe human status generated from the same JSON.
- `bodytell_voxtell_alignment_plan_2026-05-24.html`: SP-A worker plan.
- `bodytell_sampling_vocabulary_optimization_report.html`: SP-B worker plan.
- `bodytell_training_loop_eval_alignment_plan_2026-05-24.html`: SP-C worker plan.
- `runs/<TASK_ID>/PROMPT.md`: exact prompt sent to the worker.
- `runs/<TASK_ID>/RESULT.html`: worker completion report.
- `runs/<TASK_ID>/REVIEW.html`: reviewer/manager gate report.
- `runs/<TASK_ID>/GATE.html`: optional compact required-gate artifact when the reviewer separates gate evidence from `REVIEW.html`.
- `runs/<TASK_ID>/events.jsonl`: raw worker `codex --json` event stream. Do not tail this from the long-lived leader context.
- `runs/<TASK_ID>/review-events.jsonl`: raw reviewer `codex --json` event stream. Do not tail this from the long-lived leader context.
- `runs/<TASK_ID>/codex-final.md`: final Codex worker message.

## Agents

### Manager

The manager owns `lead.html` and `workflow_manifest.yaml`.

Responsibilities:

- Choose tasks whose dependencies are accepted.
- Generate a worker prompt from the manifest and the corresponding sub-plan.
- Run or assign exactly one task at a time unless a human explicitly chooses parallel execution. For checkpointed phase work, prefer `auto --max-tasks 1`.
- Review `RESULT.html`, git diff, and verification output.
- Require the phase1 Slurm gate in `REVIEW.html` before accepting a task.
- Mark the task as `accepted`, `needs_fix`, `gate_failed`, or `blocked`.
- Append accepted progress to `lead.html`.
- Use only compact status for routine monitoring: `run_manager.py status`, `workflow_status.json`, or `workflow_status.html`.

The manager must not implement SP-A/SP-B/SP-C code directly. The long-lived leader/manager context must not tail or paste `events.jsonl`, `review-events.jsonl`, full HTML reports, full diffs, or long test logs during routine monitoring.

### Reviewer

The reviewer checks worker output and writes `REVIEW.html`. The reviewer may not edit implementation files or `lead.html`.

Every reviewer must submit the phase1 Slurm gate before returning `accept`:

```bash
sbatch slurm/body-tell-phase1.sh
sacct -j <JOB_ID> --format=JobID,State,ExitCode,Elapsed -n
```

`REVIEW.html` must include the job id, final state, exit code, and a `Phase1 Slurm gate: pass` / `failed` / `blocked` field. The reviewer may return `accept` only when the main job is `COMPLETED` with exit code `0:0`. If the gate fails because of code behavior, the verdict must be `needs_fix`; if Slurm is unavailable for external cluster reasons, the verdict must be `blocked`. If a reviewer returns `accept` without gate evidence, the manager records `gate_failed`.

## General Workflow Improvements

These rules are intended to apply to every future task, not only the current SP-A work.

- Required gates are separate from suggested verification. Suggested verification catches local unit/regression issues; required gates prove the accepted work is integrated enough to advance the workflow.
- A task is not accepted until every required gate passes. For this workflow the required gate is the phase1 Slurm job.
- Gate evidence must be compact and durable: job id, command, final state, exit code, elapsed time, and the smallest relevant failure excerpt. Put it in `REVIEW.html` or `runs/<TASK_ID>/GATE.html`.
- A gate failure should be recorded as `gate_failed` when the reviewer accepted but required integration evidence is missing or failed. The task remains runnable, and the next worker prompt must include the manager note.
- Checkpointed development should use `auto --max-tasks 1` so the manager stops after one worker/reviewer cycle and a human can inspect the gate result before the next task starts.
- Commits should be made per accepted task only after required gates pass. Workflow-infrastructure changes should be committed separately from SP-A/SP-B/SP-C implementation changes.
- When Slurm exposes a deterministic failure, add or update a faster pytest/regression test in the next fix task so future workers catch the issue before the expensive gate.

## State Machine

```text
todo -> ready -> assigned -> submitted -> accepted
                              -> needs_fix
                              -> gate_failed
                              -> blocked
```

Only the manager changes status. Workers write `RESULT.html`; they do not update status themselves. A reviewer verdict of `accept` is not sufficient by itself: the manager must reject acceptance unless `REVIEW.html` contains phase1 Slurm gate evidence showing `COMPLETED` and exit code `0:0`.

## Context Budget Rules

Routine monitoring must stay under 10 lines and come from the compact status mechanism. A leader may read:

- `python Body-Tell/reports/26-05-24/run_manager.py status`
- `Body-Tell/reports/26-05-24/workflow_status.json`
- `Body-Tell/reports/26-05-24/workflow_status.html`

A leader must not run `tail`, `cat`, or unbounded reads on `events.jsonl` or `review-events.jsonl`. If a child process fails in a way that requires event-log forensics, assign a short-lived diagnostic agent or run a bounded diagnostic command outside the long-lived leader context, then bring back only the conclusion and the minimum relevant excerpt.

On `needs_fix` or `blocked`, the leader may inspect the corresponding `RESULT.html` and `REVIEW.html`, but should read targeted sections only. Full diffs and full verification logs belong in worker/reviewer artifacts, not in the leader context.

## Worker Rules

Every worker prompt must enforce these rules:

- Use only the assigned task from the assigned sub-plan.
- Read the current manager note in `PROMPT.md`; if the task is a rerun, address that note directly.
- Do not modify `lead.html` or `workflow_manifest.yaml`.
- Do not do adjacent refactors.
- Use `conda run -n voxtell ...` for verification commands.
- Write an HTML result report to the task result path.
- Include changed files, verification commands, observed outputs, blockers, and residual risks.

## Commands

List the task graph:

```bash
python Body-Tell/reports/26-05-24/run_manager.py list
```

Show runnable tasks:

```bash
python Body-Tell/reports/26-05-24/run_manager.py ready
```

Write and print compact leader-safe status:

```bash
python Body-Tell/reports/26-05-24/run_manager.py status
```

Continuously refresh compact JSON/HTML status files without reading event logs:

```bash
python Body-Tell/reports/26-05-24/run_manager.py monitor --interval 30
```

Generate a worker prompt without running Codex:

```bash
python Body-Tell/reports/26-05-24/run_manager.py prompt SP-B.Step1
```

Run a worker through Codex CLI:

```bash
python Body-Tell/reports/26-05-24/run_manager.py run SP-B.Step1
```

Run the manager loop automatically:

```bash
python Body-Tell/reports/26-05-24/run_manager.py auto --full-access
```

If Codex still asks for interactive permission, use the stronger non-interactive bypass:

```bash
python Body-Tell/reports/26-05-24/run_manager.py auto --bypass-permissions
```

To run this through one interactive leader Codex session, open Codex CLI in the repository, set full access once, then give it `LEADER_PROMPT.md`.

Generate a review prompt:

```bash
python Body-Tell/reports/26-05-24/run_manager.py review-prompt SP-B.Step1
```

Accept a reviewed task and append progress to `lead.html`:

```bash
python Body-Tell/reports/26-05-24/run_manager.py accept SP-B.Step1 --note "Dynamic sampling accepted after dataset tests passed."
```

Return a task for fixes:

```bash
python Body-Tell/reports/26-05-24/run_manager.py fail SP-B.Step1 --note "Prompt RNG test is missing cross-epoch evidence."
```

Return a task after a required integration gate fails:

```bash
python Body-Tell/reports/26-05-24/run_manager.py gate-fail SP-A.P4 --note "Phase1 Slurm gate failed with decoder/skip shape mismatch."
```

## Parallelism

The safe default is sequential execution. If parallelism is needed, run at most one task per lane at a time:

- one SP-A task
- one SP-B task
- one SP-C task

Do not run two workers that can touch the same files in the same worktree. For real parallel execution, use separate git worktrees per worker branch and merge only after manager acceptance.
