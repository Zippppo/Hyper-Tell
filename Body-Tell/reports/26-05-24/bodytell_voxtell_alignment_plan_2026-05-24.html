Body-Tell 对齐 VoxTell v1.1 实施计划

日期：2026-05-24；目标：让 Body-Tell 在结构和权重语义上尽量对齐 VoxTell v1.1，优先支持 VoxTell checkpoint 复用，而不是只做到 tensor shape 能加载。

核验结论

原差异报告中的核心不一致真实存在。 其中最影响 VoxTell 权重复用的是两类问题：一类是参数量和模块拓扑明显不同，例如 encoder/decoder；另一类是参数 shape 一致但 forward 语义不同，例如 transformer decoder。

差异点核验结果对权重复用的影响证据位置
Transformer decoder 真实存在 Body-Tell 使用 activation="gelu"，pre-norm path 执行 self-attention；VoxTell 默认 relu，active forward_pre 跳过 self-attention，只做 cross-attention 和 FFN。 参数名和 shape 可 100% 对上，但加载 VoxTell 权重后语义不等价。小模型同权重实验输出最大绝对差为 0.9258。 Body-Tell/body_tell/models/voxtell_body_model.py:69、Body-Tell/body_tell/models/transformer.py:137、VoxTell/voxtell/model/transformer.py:224
Encoder backbone 真实存在 Body-Tell 是 5-stage 手写 ConvBlock；VoxTell 是 6-stage ResidualEncoder，features 为 [32,64,128,256,320,320]，blocks 为 [1,3,4,6,6,6]。 当前 encoder 无法复用 VoxTell encoder 权重，也很可能限制性能上限。这个差异不是“轻量配置”，而是 backbone 类型不同。 Body-Tell/body_tell/models/voxtell_body_model.py:155、VoxTell/models-weight/voxtell/voxtell_v1.1/plans.json:20、VoxTell/voxtell/model/voxtell_model.py:131
Mask projections（含 num_heads） 真实存在 两层根因：(1) Body-Tell 配置写 5，但运行时被 min(num_maskformer_stages, len(encoder_channels)-1) 截成 4 级；(2) Body-Tell num_heads=32（voxtell_body_model.py:29）vs VoxTell num_heads=1（voxtell_model.py:74）。两者乘积决定实际 projection 输出。 Body-Tell 实际输出 [32, 2048, 4096, 8192]（channels × 32，stage 0 例外）；VoxTell 实际输出 [32, 64, 128, 256, 320] 量级（channels × 1）。原报告写 VoxTell 最高级 10240 是错误的。 即使 encoder 升到 6 级、stage 截断消除，num_heads 不同会导致 project_to_decoder_channels.* 与下游 mask fusion 模块的 shape 不匹配，权重无法直接复用。 Body-Tell/body_tell/models/voxtell_body_model.py:29、:83、:89；VoxTell/voxtell/model/voxtell_model.py:74、:177、:370
Decoder 真实存在 Body-Tell 是 4-stage 手写 decoder，GroupNorm + GELU；VoxTell 是 5-stage VoxTellDecoder，使用 StackedConvBlocks 并继承 encoder 的 InstanceNorm3d + LeakyReLU + conv_bias=true。 当前 decoder 无法复用 VoxTell decoder 权重。VoxTell 还总是构造 seg_layers，用于 checkpoint 兼容。 Body-Tell/body_tell/models/voxtell_body_model.py:174、VoxTell/voxtell/model/voxtell_model.py:280
Position / memory 排列 真实存在 Body-Tell 动态生成 D,H,W 顺序的位置编码；VoxTell 使用 init 时固定的 pos_embed buffer，并用 H,W,D 顺序 flatten。 Body-Tell 当前 volume_size=[129,108,256]（见 configs/phase1_voxtell_body.yaml:25，2026-05-24 commit 5c31e57 后裁剪到此尺寸）不是 192³，直接加载 VoxTell pos_embed 会遇到 shape/语义问题。需要明确动态重建或改成 192³ profile。 Body-Tell/body_tell/models/voxtell_body_model.py:118、VoxTell/voxtell/model/voxtell_model.py:185、:229

目标状态

新增一个 VoxTell-aligned 架构路径，而不是在现有轻量模型上做零散补丁。该路径的 state dict 命名应尽量贴近 VoxTell，降低权重加载映射复杂度。

模块目标实现权重复用要求
encoder 使用 dynamic_network_architectures.building_blocks.residual_encoders.ResidualEncoder，参数与 VoxTell v1.1 plan 一致。 加载 VoxTell encoder.* 权重，shape mismatch 必须报错。
decoder 移植 VoxTell VoxTellDecoder，保留 5 个 stage、StackedConvBlocks、seg_layers always-built 语义。 加载 VoxTell decoder.* 权重；为了严格兼容，可保留 self.encoder = encoder 这一共享引用。
transformer_decoder 与 VoxTell 的 TransformerDecoderLayer 行为一致：默认 relu，pre-norm 不执行 self-attention。 同名加载 VoxTell transformer 权重；保留 self-attention 参数，即使 active path 不使用。
project_bottleneck_embed 使用 VoxTell 命名，替代当前 project_visual_embed；结构仍为 320 -> 2048 -> 2048。 直接加载 VoxTell project_bottleneck_embed.*。
project_text_embed 保持 2560 -> 2048 -> 2048。 直接加载 VoxTell project_text_embed.*。
project_to_decoder_channels + num_heads 使用 VoxTell 命名，替代当前 mask_projections；5 级；输出维度 = encoder 各级 channels × num_heads。条件性对齐（父表 #13，域差异豁免）：如果实验目标是直接复用 VoxTell project_to_decoder_channels.* 和 mask fusion 权重，则必须设 num_heads=1，实际输出 [32, 64, 128, 256, 320] 量级；如果仅做结构对齐而不复用这部分权重（更符合 Body-Tell occupancy grid 任务的需求，因为 num_heads=32 提供 32× mask fusion 容量是为该任务调过的），则保留 num_heads=32。 选 num_heads=1：直接加载 VoxTell project_to_decoder_channels.*。选保留 num_heads=32：必须把 project_to_decoder_channels.* 与 mask fusion 相关 key 全部加入跳过白名单，并在 audit 中登记"为保留 Body-Tell mask fusion 容量而跳过这部分权重复用"。
pos_embed 提供两种 policy：dynamic_hwd 用于当前非 192³ Body-Tell 输入；fixed_voxtell_192 用于严格复刻 VoxTell patch。 dynamic_hwd 模式跳过 checkpoint 中 pos_embed；fixed_voxtell_192 模式要求 shape 完全匹配。

实施阶段

P0：建立兼容性基线

  1. （2026-05-24 新增）num_heads 与 projection shape 显式审计：新增 Body-Tell/scripts/audit_voxtell_alignment.py（如果与已有 scripts/parameter_audit.py 重合，应共享 utilities）。脚本第一步必须打印两侧 num_heads 值和每级 project_to_decoder_channels 实际输出 shape，确认 VoxTell 为 num_heads=1 + 输出 [32, 64, 128, 256, 320] 量级，Body-Tell 当前为 num_heads=32 + 输出 [32, 2048, 4096, 8192]。该审计是父表 #13 num_heads 决议（路径 A vs 路径 B）的输入材料，audit 完成后才能在 P4 选定路径。
  2. 实例化 Body-Tell aligned model 和 VoxTell v1.1 model，输出总参数、模块参数、state dict key/shape 覆盖率。
  3. 新增 transformer golden test Body-Tell/tests/test_voxtell_transformer_compat.py：同一小尺寸 state dict、相同输入、dropout=0 时，Body-Tell aligned transformer 输出必须与 VoxTell transformer 完全一致或在 1e-6 内。
  4. 新增 weight key audit：读取 VoxTell/models-weight/voxtell/voxtell_v1.1/fold_0/checkpoint_final.pth（已确认文件存在）的 network_weights，检查除 pos_embed 外的目标 key 是否全部可加载。

验收：当前轻量模型仍可跑旧测试；新增审计能明确报告哪些 key 会加载、哪些 key 因 policy 跳过；且 num_heads 决议（设为 1 / 保留 32）必须先于 P4 落地。

P1：先修 transformer decoder 语义

  1. 把 Body-Tell/body_tell/models/transformer.py 的 active forward_pre 改为 VoxTell 版本：从 norm2 -> cross_attn -> norm3 -> FFN 开始，不执行 self-attention。
  2. 构造层时去掉 activation="gelu"，或显式设为 relu，与 VoxTell 默认一致。
  3. 保留 self_attn、norm1 等参数名和 shape，保证 checkpoint key 不变。

验收：Body-Tell aligned transformer 与 VoxTell transformer 小模型 parity test 通过；加载 VoxTell transformer_decoder.* 后无 missing/unexpected/shape mismatch。

P2：替换 encoder 为 VoxTell residual encoder

  1. 新增配置文件 Body-Tell/configs/phase1_voxtell_aligned.yaml，设置 encoder_channels: [32,64,128,256,320,320]、n_blocks_per_stage: [1,3,4,6,6,6]、decoder_layer: 4、num_maskformer_stages: 5。
  2. 在模型配置中加入 architecture: voxtell_v1_1 或 backbone: residual_encoder，让旧轻量模型和 aligned 模型可并存。
  3. 使用 VoxTell 相同构造参数：conv_bias=true、InstanceNorm3d(eps=1e-5, affine=True)、LeakyReLU(inplace=True)、BasicBlockD、return_skips=True、disable_default_stem=False。
  4. 把依赖写入 Body-Tell 环境声明：dynamic-network-architectures>=0.4.1,<0.5 和 einops。若采用 VoxTell 原始位置编码实现，再加入 positional-encodings。

验收：encoder 参数约 90.30M；输出 6 个 skip；encoder.output_channels 和 encoder.strides 与 VoxTell 一致；VoxTell encoder.* 权重可加载。

P3：替换 decoder 为 VoxTellDecoder

  1. 移植 VoxTellDecoder 到 Body-Tell 模型代码或独立模块，优先保持源码结构、字段名和 state dict key 兼容。
  2. 使用 n_conv_per_stage_decoder=[1,1,1,1,1]，并继承 encoder 的 conv/norm/nonlin/dropout/bias。
  3. 即使 deep_supervision=false，也总是构造 5 个 seg_layers，与 VoxTell checkpoint 兼容。
  4. 移除 Body-Tell 当前 decoder 中 upconv 后的自动 F.interpolate fallback；如果输入尺寸不能通过转置卷积对齐，应在测试中显式暴露，或在数据裁剪 policy 中解决。

验收：decoder 本体参数约 12.18M；5 个 stage / 5 个 transposed conv / 5 个 seg layer；VoxTell decoder.* 权重可加载。统计 decoder 参数时需扣除共享 encoder，避免误读为 102.47M。

P4：对齐 projection、mask head、num_heads 决议和 forward 排列

P4 num_heads 决议（父表 #13，域差异豁免）：在执行 P4 前必须先选定路径——
  • 路径 A（强对齐，复用权重）：num_heads=1，projection 输出 [32, 64, 128, 256, 320]，可直接加载 VoxTell project_to_decoder_channels.* 与 mask fusion 权重。代价：失去 Body-Tell 现有 32× mask fusion 容量。
  • 路径 B（结构对齐但放弃这部分权重复用，推荐）：保留 num_heads=32，projection 输出维持 [32, 2048, 4096, 8192, 10240]（encoder 升 6 级后扩展），project_to_decoder_channels.* 与 mask fusion 权重 不加载，加入 P6 跳过白名单。理由：Body-Tell 处理 occupancy grid 与 VoxTell 医学影像任务差异显著，这部分权重的可迁移性弱，而 mask fusion 容量对当前任务更有价值。
默认建议路径 B，并在 P0 audit 中显式打印决议来源。如果 audit 没有给出明确收益对比，保持现状（路径 B）即可，不强制对齐 num_heads。
  1. （2026-05-24 修订）num_heads 决议生效：按上方选定路径执行。路径 A 需把配置 num_heads 默认值改为 1，同时审计 Body-Tell 中所有依赖 num_heads 的 reshape（如 view(B, num_heads, C//num_heads)）和 decoder stage 内 input_features + num_heads 的通道拼接是否正确退化；路径 B 不改 num_heads，但需要在 P6 把 project_to_decoder_channels.* 与 mask fusion 相关 key 加入跳过白名单。
  2. 把当前 project_visual_embed 改为 VoxTell 命名 project_bottleneck_embed；把 mask_projections 改为 project_to_decoder_channels。
  3. encoder 变为 6-stage 后，projection stage 自然恢复为 5 级，不再用 len(encoder_channels)-1 隐式截断配置。
  4. forward 中使用 VoxTell 的 memory 排列：b c d h w -> b h w d c -> (h w d) b c。
  5. decoder 输出仍保持 Body-Tell 训练和推理需要的 (B, N, D, H, W) 语义；如果内部沿用 VoxTell 注释中的 h,w,d 命名，需要在单测里用非立方输入验证轴没有被误换。

验收：5 个 projection 输出维度按选定路径（路径 A: [32, 64, 128, 256, 320]；路径 B: [32, 2048, 4096, 8192, 10240]）；非立方输入的输出 shape 与 target mask 完全一致；mask fusion 模块在选定 num_heads 下有显式单测；audit 报告记录选定路径与原因。

P5：位置编码策略

策略适用场景实现方式权重加载
dynamic_hwd 推荐用于当前 Body-Tell 数据，因为 [129,108,256]（commit 5c31e57 后的实际 crop 尺寸）不是 VoxTell 的 192³ patch。 按 selected feature 的实时 H,W,D 生成 deterministic 3D positional encoding，并缓存到 device/dtype。 跳过 VoxTell checkpoint 的 pos_embed，其余权重严格加载。
fixed_voxtell_192 用于做完全复刻实验，或想验证整份 VoxTell state dict 可严格加载。 把 Body-Tell 数据裁剪/填充到 [192,192,192]，注册 shape 为 (1728,1,2048) 的 pos_embed。 要求 checkpoint 中 pos_embed 直接加载；数据体素尺寸也必须配套调整。

建议先落地 dynamic_hwd。它牺牲的是固定 buffer 的逐元素复用，不牺牲 transformer 权重语义；对当前非立方 Body-Tell 数据更可实现。

P6：权重加载入口

  1. 新增训练参数 --init-from-voxtell VoxTell/models-weight/voxtell/voxtell_v1.1，在 optimizer 构造前加载。
  2. 新增 loader：读取 fold_0/checkpoint_final.pth 的 network_weights，按当前 aligned 模型 key 直接加载。
  3. （2026-05-24 修订）显式 unexpected_keys 白名单：调用 model.load_state_dict(state_dict, strict=False) 后，断言 set(unexpected_keys) ⊆ POLICY_ALLOWED_SKIP（默认 {"pos_embed"}；若 P4 决定保留 num_heads=32，再扩展到 {"pos_embed", "project_to_decoder_channels.*", "decoder."}）。任何 shape mismatch 必须抛错，不得静默跳过；missing_keys 不为空时打印完整列表并要求显式确认。
  4. 日志记录 loaded tensor 数、loaded 参数数、skipped key、missing key、unexpected key，并写入训练 checkpoint 的 config.init_from_voxtell。
  5. 不要把旧 Body-Tell checkpoint 与 aligned 模型静默混用。旧 checkpoint 应只用于旧架构；aligned checkpoint 应带 architecture: voxtell_v1_1。

验收：在 CPU 上能完成权重加载审计；unexpected_keys 与 policy 白名单一致；在 GPU 上能完成至少 1 个 batch 的 forward/loss/backward smoke。

P7：训练实验矩阵

实验初始化目的建议
A：当前基线 现有轻量 Body-Tell checkpoint 保持可回退基线。 记录当前 val foreground Dice 约 0.3257。
B：aligned random VoxTell-aligned 结构，随机初始化 分离“结构变大”与“VoxTell 权重”的收益。 先跑短程，确认显存和 loss 曲线。
C：aligned prompt-only 只加载 transformer / text / projection 权重 评估 prompt decoder 复用本身的收益。 可作为中间回退，因为 encoder 输入域差异最大。
D：aligned full 加载 encoder、decoder、projection、transformer；按 policy 跳过或加载 pos_embed 完整复用 VoxTell 权重，验证性能上限。 使用分组学习率：prompt/transformer 较低，encoder/decoder 可稍高；不要一开始冻结全部 encoder。

主要风险

风险影响控制方式
显存和训练成本上升 模型从 328.98M 增至约 440.03M，encoder activation 也明显变大。 先用 batch=1、AMP、短程 smoke；必要时加入 gradient checkpointing 或降低 prompt batch。
VoxTell encoder 输入域是医学影像，Body-Tell 是 occupancy grid 完整加载 encoder 不一定优于随机初始化，甚至可能初期收敛慢。 保留 random / prompt-only / full 三组 A/B；使用分组学习率，不把 full 权重复用当作唯一方案。
非 192³ 输入与 fixed pos_embed 冲突 严格加载整份 VoxTell state dict 会失败或语义错误。 默认 dynamic_hwd 并明确跳过 pos_embed；另建 fixed_voxtell_192 profile 做完整复刻。
旧 Body-Tell checkpoint 与新结构不兼容 推理/恢复训练可能 silent mismatch 或 strict load fail。 checkpoint 中记录 architecture；inference strict 校验；不给旧 checkpoint 自动迁移到 aligned 结构。
直接复制 VoxTell decoder 会引入共享 encoder 引用 参数统计容易误读，state dict 可能包含共享模块路径。 为了权重兼容优先保留 VoxTell 结构；审计脚本单独报告 decoder-own 参数。

建议优先级

  1. （2026-05-24 上调）P0 num_heads 与 projection shape 审计。这是父表 #13 num_heads 决议的输入材料；不在 P4 落地前完成审计，P3/P4/P6 都会建立在未经确认的输出维度假设上。允许 audit 结论是"保留 num_heads=32"（路径 B），但必须显式记录。
  2. P1 transformer 语义对齐。 当前最危险的不一致，因为 shape 能加载会掩盖语义错误。
  3. 随后做 P2/P3 encoder 和 decoder。 当前 encoder 参数差距过大且结构不同，性能问题很可能与此相关；若目标是对齐 VoxTell，不应继续保留轻量自定义 backbone 作为主路线。
  4. P5 位置编码要在权重加载前定策略。 当前 Body-Tell 非立方 volume（[129,108,256]）不适合直接复用 VoxTell 固定 pos_embed。
  5. 最后才跑完整训练实验。 没有 architecture/weight audit 通过前，不建议直接用 VoxTell checkpoint warm-start 长训。
