结论

Body-Tell 的 Python 代码基本可以作为独立项目迁出；没有发现训练/推理核心代码依赖 Body-Tell/ 外部的 Python 脚本。

当前主要问题不是跨目录 import，而是运行入口和默认路径仍假设它位于 Hyper-Tell/Body-Tell 这个父子目录结构中。也就是说：代码边界已经比较干净，但还没有达到“拷贝 Body-Tell/ 后从目录内部直接训练”的状态。

路径可移植性验证

从 Body-Tell/ 目录内部读取当前训练配置，并按 data.root 检查训练所需路径，全部解析失败：

Body-Tell/configs/label_vocab.json: False
Body-Tell/Dataset/dataset_split.json: False
Body-Tell/artifacts/data_stats/class_presence.json: False
Body-Tell/artifacts/text_embeddings/prompt_embeddings.pt: False
Body-Tell/Dataset/voxel_data: False

原因是当前配置要求工作目录为 Hyper-Tell/ 父目录，而不是 Body-Tell/ 自身。这与 train.py 顶部用法说明不一致。

inference.py 的路径处理更接近独立项目：它用 ROOT = Path(__file__).resolve().parent 设置默认配置和 checkpoint，并通过 resolve_body_root() 兼容 data.root: Body-Tell。

  1. 把训练配置的项目根处理统一到 train.py：相对 data.root 优先按配置文件所在目录解析；当值为 Body-Tell 且脚本根同名时回退到脚本根。可直接复用 inference.py::resolve_body_root() 的思路。
  2. 新增 Body-Tell/slurm/body-tell-phase1.sh，脚本内部定位 Body-Tell 根目录，避免硬编码 /home/comp/csrkzhu/code/Hyper-Tell。
  3. 把推荐启动命令改成项目内命令：cd Body-Tell && torchrun --nproc_per_node=2 train.py --config configs/phase1_voxtell_body.yaml --checkpoint-dir checkpoints/phase1_bz2。
  4. 新增环境声明文件，例如 requirements.txt 或 environment.yml。当前 source activate voxtell 只是机器本地约定。
  5. 新增迁移检查命令：验证 Dataset/dataset_split.json、Dataset/voxel_data、configs/label_vocab.json、artifacts/data_stats/class_presence.json、artifacts/text_embeddings/prompt_embeddings.pt 是否存在。
  6. 把 standalone CI/smoke test 固化为：cd Body-Tell && PYTHONPATH=. pytest tests -q，并增加一个只构建 dataset 的路径测试。

最终判断

Python 代码独立性：通过 没有发现 Body-Tell/ 依赖外部 Python 脚本。

项目迁移就绪度：部分通过 如果只是拷贝代码和数据，核心包可运行；如果希望像独立项目一样从目录内部训练，还需要修复训练配置根目录、迁移 Slurm 入口和补齐环境声明。

当前最大风险 slurm/body-tell-phase1.sh 和 data.root: Body-Tell 让训练依赖父仓库工作目录。这个风险会在迁出时直接暴露。
