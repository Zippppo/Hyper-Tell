一句话结论

不是只有之前提到的几个问题。 从 Python import 角度看，Body-Tell/ 没有发现依赖外部本地 Python 脚本；但从“把目录单独迁出并继续训练/推理/复现实验”的角度看，目前至少有 12 类迁移问题。

真正阻断迁移的不是 body_tell 包内部代码，而是启动入口、配置根目录、git/ignore 状态、checkpoint 内嵌配置、环境声明和 artifact 交付方式。

审计边界

  • 检查 Body-Tell/ 内 Python import，确认是否引用父仓库 models/、utils/、data/、eval/、scripts/、VoxTell/voxtell。
  • 检查训练、推理、可视化、artifact 脚本、Slurm 和配置中的路径假设。
  • 检查 .gitignore 和 git tracked files，判断迁出方式是“复制目录”还是“从 git clone 重建”时会缺什么。
  • 没有执行完整 GPU/DDP 训练；本报告关注迁移可运行性和复现风险。

正向结论

核心包边界是干净的。 AST import 扫描没有发现 Body-Tell/ 内 Python 代码 import 父仓库本地模块。非标准库依赖只有第三方包：

numpy

因此，如果把代码、数据、artifact、checkpoint 和环境一起迁出，Body-Tell/ 可以成为独立项目。当前问题集中在项目工程化边界。

完整迁移问题清单

ID 级别 问题 证据 迁移影响 建议处理
M01 阻断训练 训练 Slurm 入口在 Body-Tell/ 外部，并硬编码父仓库绝对路径。 slurm/body-tell-phase1.sh:14 使用 cd /home/comp/csrkzhu/code/Hyper-Tell；:16-17 使用 Body-Tell/train.py 和 Body-Tell/configs/...。 只迁出 Body-Tell/ 后，原训练入口不存在且路径失效。 新增 Body-Tell/slurm/body-tell-phase1.sh，用脚本自身定位项目根，避免绝对路径。
M02 阻断训练 训练配置的 data.root 与 standalone 工作目录矛盾。 Body-Tell/configs/phase1_voxtell_body.yaml:18 是 root: Body-Tell；train.py:5-9 的说明却是从 Body-Tell/ 内运行 python train.py --config configs/...。 从 Body-Tell/ 目录内运行时，数据路径会解析成 Body-Tell/Body-Tell/...。 把相对 data.root 解析到配置文件所在项目根；或迁出前把配置改成 root: . 并同步 checkpoint 配置。
M03 阻断训练 train.py 不做项目根解析，只把 cfg["data"]["root"] 原样交给 Dataset。 train.py:344 加载配置；train.py:361 调用 build_dataset；body_tell/data/dataset.py:42-64 从 root 派生所有数据路径。 训练能跑依赖当前调用者的工作目录，而不是项目自身位置。 在 train.py 中加入与 inference.py::resolve_body_root() 等价的解析逻辑。
M04 阻断推理 checkpoint 内嵌旧配置，推理默认优先使用 checkpoint 配置。 train.py:452-460 保存 checkpoint 时写入 "config": cfg；inference.py:727-730 如果 checkpoint 有 config 就覆盖 CLI 配置。 即使迁出后修了 configs/phase1_voxtell_body.yaml，旧 checkpoint 仍可能把 data.root: Body-Tell 带回推理流程。 迁移 checkpoint 时重写内嵌 config，或推理时默认只采用 checkpoint 的 model 配置，data root 从 CLI/config 文件解析；短期可用 --ignore-checkpoint-config。
M05 阻断复现 关键数据、embedding cache 和 checkpoint 被 git ignore，不会随普通 git clone 迁移。 .gitignore:8-9 忽略 Body-Tell/Dataset/ 和 Body-Tell/artifacts/；.gitignore:101-112 忽略 *.npz 和 *.pt；实际 git 跟踪 .npz=0、.pt=0。 git clone 后缺少 10,779 个 voxel .npz、prompt_embeddings.pt 和 best.pt，训练/推理都会失败。 定义迁移包清单：代码走 git，数据/artifact/checkpoint 走 rsync、对象存储或 release asset，并提供校验脚本。
M06 阻断推理 默认推理 checkpoint 路径指向未跟踪文件。 inference.py:59-62 默认 ROOT/checkpoints/phase1_bz2/best.pt；实际文件大小约 3.95GB，且 Body-Tell/checkpoints/ 被 ignore。 迁出代码但不复制 checkpoint 时，默认推理命令无法运行。 在 README/迁移脚本中把 checkpoint 作为必需 artifact；或让推理入口在缺失时给出下载/放置路径提示。
M07 阻断训练 prompt embedding cache 是训练必需文件，但未纳入可复现交付。 body_tell/data/dataset.py:75-82 初始化 Dataset 时加载 prompt_embeddings.pt；该文件位于 Body-Tell/artifacts/text_embeddings/，被 ignore。 迁出代码后，Dataset 初始化会直接失败，除非复制 cache 或重跑 embedding 构建。 把 prompt_embeddings.pt 纳入 artifact 清单，并记录 Qwen 模型、hash 和重建命令。
M08 复现风险 embedding cache 与 label_vocab.json 严格 hash 绑定。 body_tell/data/vocabulary.py:281-292 检查 prompt records 和 vocab hash；不一致会报错。 迁移中如果格式化、修改或重新生成 label_vocab.json，旧 cache 会失效。 迁移时把 vocab 和 cache 成对校验；修改 vocab 后必须重建 cache。
M09 环境风险 Body-Tell/ 没有独立环境声明文件。 未发现 requirements.txt、environment.yml、pyproject.toml 或 setup.py；slurm/body-tell-phase1.sh:12 依赖 source activate voxtell。 迁到新机器后不知道该装哪些版本，尤其是 torch、transformers、wandb、plotly、CUDA 组合。 新增 environment.yml 或 requirements.txt，区分 train、inference、visualization、dev/test extras。
M10 工具风险 可视化脚本自动寻找原始数据时假设固定输出目录深度。 visualize_predictions.py:398-402 假设 Body-Tell/outputs/.../CASE_ID/CASE_ID_predictions.npz，用 input_path.parents[2] 推导项目根。 迁移后如果输出目录放到项目外、目录层级变更或重命名 run，自动加载 original data 会失败。 保留 --original 作为显式路径；或新增 --body-root 参数替代 parents[2]。
M11 低风险清理 报告和 metadata 中仍有 Body-Tell/Dataset/...、Hyper-Tell 或 parent root 概念。 scripts/scan_class_presence.py:175-180、configs/label_vocab.json、artifacts/data_stats/class_presence.json 中有来源字符串；scripts/parameter_audit.py:22 定义 REPO_ROOT = parents[2]。 多数不是运行路径，但迁出后容易误导后续维护者，以为仍依赖 Hyper-Tell。 重生成 artifact metadata 或把字段命名成 source_note；删除/弱化父仓库 metadata。
M12 项目化不足 没有安装式包入口，复用依赖当前目录或 PYTHONPATH。 测试依赖 conftest.py 注入 Body-Tell 根；脚本通过自身路径插入 sys.path。这能运行，但不是标准 package 分发方式。 迁出后如果用户用 notebook、外部脚本或调度系统从任意目录 import，可能需要手动设置 PYTHONPATH。 新增 pyproject.toml，支持 pip install -e .，并提供 console scripts。
