问题 1: inside_body_empty 进入词表

需要重点复查: Body-Tell/configs/label_vocab.json 中保留了 id=0 / source_name=inside_body_empty / canonical=inside body empty space。 该项虽然被标记为不作为正样本训练、不计入 foreground eval, 但它仍拥有 prompts, 因此可能进入 prompt records、embedding cache 或后续推理候选集合。

问题 1 当前配置片段

{
  "id": 0,
  "source_name": "inside_body_empty",
  "canonical": "inside body empty space",
  "prompts": [
    "inside body empty space",
    "empty unlabeled body interior",
    "non-organ space inside the body"
  ],
  "train_as_positive": false,
  "eval_as_foreground": false,
  "notes": "Background/empty interior label; excluded from foreground training and foreground mean Dice."
}

问题 1 为什么可疑

风险点说明
语义边界不清 inside_body_empty 更像体内未标注/非器官空间, 不一定等价于普通背景, 也不一定适合作为可查询 anatomical prompt。
词表污染 即使 train_as_positive=false, 只要 prompts 被 flatten 或缓存 embedding, 后续开放 prompt 推理和候选类别枚举就可能误用它。
训练路径隐患 需要确认 sampler、target mask 构造、collate、loss、metric 全链路都严格遵守 ignore 语义, 没有把 class_id=0 当作普通 negative 或 foreground。
评估路径隐患 需要确认 foreground mean Dice、canonical prompt eval、negative false-positive rate 和任何 inference analysis 都不会把该类计入目标结构。

问题 2: Label alternatives 扩展策略不符合预期

需要重点复查: 正确的 label vocabulary 扩展思路应该同时覆盖 single-label alternatives、combined-label alternatives, 并满足每个 concept 至少 5 个经过验证的 variants。当前 label_vocab.json 更像是为每个基础类写了少量 prompts, 再手工加入少量 aggregate, 与这个扩展契约不一致。
期望契约例子当前风险
single-label alternatives 每个基础 label 应生成多种医学表达。例如 kidney 可以扩展为 renal parenchyma、renal organ、kidney tissue 等。 当前很多基础 label 只有 3 个 prompts, 且更像模板化同义词, 不足以覆盖临床/解剖表达的真实变体。
combined-label alternatives 生成临床上有意义的组合标签。例如 left rib 1 到 left rib 12 可以组合成 left rib cage; left kidney 和 right kidney 可以组合成 kidneys 或 bilateral kidneys。 当前 aggregate concepts 数量和覆盖范围有限, 可能遗漏许多临床上自然查询的组合概念。
至少 5 个 validated variants per concept 附录 C 要求每个 concept 至少生成 5 个经过验证的变体。 当前基础类通常 3 个 prompts、rib 类约 4 个 prompts、aggregate 通常 3 个 prompts, 没有达到至少 5 个 validated variants 的要求。

问题 2 为什么可疑

风险点说明
Prompt coverage 不足 如果每个 concept 的语言变体不足, 模型可能只学到少量模板表达, 对真实医学查询、临床缩写或组织学表达泛化较弱。
组合概念不完整 只覆盖少量 aggregate 会让模型缺少自然组合查询能力, 例如 rib cage、bilateral organs、organ systems 或解剖区域级查询。
Validated variants 缺失 当前 prompts 看起来没有显式记录人工或规则验证状态, 无法证明每个 concept 已满足“至少 5 个经过验证的变体”。
训练目标与文档契约漂移 如果附录 C 是设计契约, 当前词表会导致训练数据、embedding cache、评估 prompt set 都建立在不完整的 label expansion 上。

后续审计清单

  • Label semantics 回到 Body-Tell/Dataset/dataset_info.json 和原始 voxel_labels, 确认 label 0 的真实含义。
  • Vocabulary contract 判断非 foreground / ignore 类是否应该完全移出 prompt records 和 embedding cache, 还是仅作为 metadata 保留。
  • Sampler 检查 positive/negative prompt 采样是否可能选择 class_id=0。
  • Mask target 检查 target mask 构造是否可能为 inside_body_empty 生成监督 mask。
  • Evaluation 检查所有评估入口是否只用 eval_as_foreground=true 的类别。
  • Single-label variants 为每个基础 label 审计医学表达覆盖, 例如 kidney/renal parenchyma/renal organ/kidney tissue 等是否完整且 validated。
  • Combined-label variants 系统性审计临床有意义的组合标签, 例如 left rib cage、kidneys、bilateral kidneys 等是否进入 concepts。
  • Variant count 检查每个 concept 是否满足至少 5 个 validated variants; 不满足时需要扩展词表并重建 embedding cache。
  • Decision 如果确认是逻辑 bug, 后续应明确修复策略: 删除 prompts、重建 embedding cache、更新测试并补充 regression test。

当前建议

在完成审计前, 暂时把 inside_body_empty 和 label alternatives 扩展不足都视为高优先级潜在 bug。 不应默认把 inside_body_empty 作为合法 anatomical query、foreground 类或可解释的模型输出目标; 也不应默认当前 label_vocab.json 已经满足附录 C 的 variant 覆盖要求。
