VoxTell：自然语言如何驱动 3D 医学分割

面向后续改进的源码级解读 · 基于 voxtell/ v1.1 · 2026-05-19

1. 一句话回答

VoxTell 把体素图像和文本提示分别编码到同一隐空间， 然后用一个 Transformer 解码器让文本作为 query 去“查询”图像 bottleneck 特征（memory）， 得到每条 prompt 一个 mask embedding； 该 embedding 在 U-Net 解码器的 多层尺度上以 einsum 与特征图做逐通道点乘， 直接生成对应该 prompt 的分割 logits。文本是「条件」，但条件作用方式是 MaskFormer 风格的动态 1×1×1 卷积核。

3. 两路编码：图像与文本

① 图像编码（3D ResidualEncoder）

nnU-Net 风格的 6 阶残差卷积主干（来自 dynamic_network_architectures）， 输入 patch 大小 192³，通道 [32, 64, 128, 256, 320, 320]， 每阶下采样 2×。
输出 6 个尺度的 skip 特征图，最深的 stage 4 bottleneck（12³×320） 会被作为 Transformer 的 memory。

② 文本编码（冻结的 Qwen3-Embedding-4B）

每条 prompt 都被 指令包裹（text_embedding.py:14）：
"Instruct: Given an anatomical term query, retrieve the precise anatomical entity and location it represents\nQuery: {prompt}"
Tokenize → 过 Qwen3-Embedding-4B → 取最后一个非 padding token 的 hidden state， 得到 2560 维向量，每个 prompt 一个。

关键点：文本编码器是 frozen 的，只在推理时把模型搬到 GPU 算 embedding，算完立刻 to('cpu') 让位给 3D 主干（predictor.py:189-205）。文本对图像的「条件作用」全部由下面那个小 Transformer + 投影头来学习。

4. 融合核心：Prompt Decoder（Transformer 6 层）

定义在 voxtell_model.py:191-202，基于 DETR 风格的 TransformerDecoder：

角色张量形状（推理时）来源
query (tgt)文本 embedding(N, 1, 2048)Qwen3 输出 → 2048 投影
memory图像 bottleneck(12·12·12, 1, 2048)encoder stage 4 → 2048 投影
pos embed3D 正弦位置编码(12³, 1, 2048)注册为 buffer（voxtell_model.py:187）

6 层 cross-attention 之后，每条 prompt 收敛成一个 2048-d 的「mask embedding」。 这是文本条件信息的最终载体，它将告诉解码器“你要找的是肝、不是脾”。

4.1 把一个向量变成多尺度的卷积核

同一个 2048-d mask embedding 会被 5 个独立 MLP（project_to_decoder_channels，voxtell_model.py:171-183） 投影到解码器 5 个尺度的通道数上：

stage 0 (192³) ← 32 ch stage 1 ( 96³) ← 64 · num_heads(=32) stage 2 ( 48³) ← 128 · 32 stage 3 ( 24³) ← 256 · 32 stage 4 ( 12³) ← 320 · 32

这相当于把单个 embedding 扩展成 5 个不同分辨率上的「动态滤波器」。

5. Image Decoder：MaskFormer 风格融合

在 VoxTellDecoder.forward()（voxtell_model.py:405-472）， 解码器从底向上逐级 upsample 并 concat skip。在有 mask embedding 的阶段，做两件事：

5.1 中间层：作为 num_heads 个额外通道注入

fusion = einsum('b c h w d, b n c -> b n h w d',
                feat, mask_embed.view(B, num_heads=32, C//32))
feat   = torch.cat([feat, fusion], dim=1)   # 把 32 个 head 当 32 个新通道

每个 head 是 该 prompt 的一个加权空间响应图，与原始特征拼接后送入下一阶 conv block —— 文本条件就这样“渗透”进每一层解码。

5.2 最末层：直接生成分割 logits

seg = einsum('b c h w d, b n c -> b n h w d', feat, mask_embed)
# feat: 高分辨率特征 (B, C, X, Y, Z)
# mask_embed: 每条 prompt 一个长度为 C 的向量 → 等价于动态 1×1×1 卷积核

这就是 MaskFormer/MaskDINO 的经典做法：把分类 head 换成内积， 每条 prompt 用自己专属的向量去和像素特征做点积，正值即前景。

为什么这种结构能泛化到「未见过的文本」？
prompt 端走的是冻结的大语言模型嵌入空间——语义近邻自然就近邻； 图像端只需要学会“把图像特征拉到同一个语义空间”， 而不是“给每个解剖学类别学一个分类权重”。一切新概念都被表示为同一隐空间里的一个向量。

6. 推理工程：滑窗 + 高斯加权 + 流水线

  • 预处理（predictor.py:110）：RAS 重定向 → 非零裁剪 → Z-score。不做 spacing resample。
  • 滑窗（predictor.py:208）：patch 192³，步长 50% 重叠，对小图自动 zero-pad。
  • 高斯加权：每个 patch 输出乘以中心权重高的 3D Gaussian，再除以累计权重，避免拼接边界伪影。
  • 生产者-消费者线程：Thread 加载下一块 patch 与 GPU 推理并行（predictor.py:281-297）。
  • 显存友好：torch.autocast + half-precision logits 累加； 每条 prompt 都得到一个完整体积 logit，但 N 通常不大。
  • 后处理：sigmoid > 0.5 得到独立二值掩膜；最后 insert_crop_into_image 还原到原始尺寸。

8. 想动手改造？关注这些「可调插口」

替换文本编码器

改 VoxTellPredictor.__init__ 的 text_encoding_model 与 构造时的 text_embedding_dim（当前 2560，对应 Qwen3-4B）。 投影头 project_text_embed 自动适配新维度。

条件注入策略

核心在 VoxTellDecoder.forward 的两次 einsum。 想试 FiLM、cross-attention、或可学习偏置？这里替换即可， num_heads/num_maskformer_stages 控制注入的宽度和深度。

Prompt 模板

wrap_with_instruction() 的 Instruct 串影响 Qwen embedding 的语义聚焦。 想做病变/形态/位置等不同维度提示，可在此分流。

选哪一层做 memory

decoder_layer（默认 4）决定 Prompt Decoder 看哪层 encoder 特征。 浅层（更高分辨率）可能改善小目标但显存暴涨； DECODER_CONFIGS 已枚举 0–5 的形状/通道。

⚠️ 改造前必读：
① 输入必须 RAS 朝向；否则左右肾、左右肺会全部错乱（README 也强调这一点）。
② 模型没有显式 spacing 归一化，极端体素间距会导致性能下降。
③ 训练权重与 plans.json 的 arch_kwargs 强耦合，改动主干结构需重训。
